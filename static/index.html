Como Funciona o CROM SQL Delivery?

Entenda a engenharia por trás do carregamento instantâneo de imagens e economia extrema de largura de banda.

1. Divisão Espacial e Codificação por Índices

O sistema divide a imagem em blocos espaciais não sobrepostos (de 4x4 pixels no codebook ativo). Cada bloco contém 16 pixels com canais de cores Vermelho, Verde e Azul (RGB), totalizando 48 dimensões de dados de cor.

Em vez de transmitir os 48 bytes de cor brutos para cada bloco, o codificador consulta o Dicionário Visual ("Cérebro") para encontrar o padrão mais parecido e transmite apenas o seu índice de 16 bits (2 bytes).

Taxa de Compressão Estática por Bloco: 24x de Economia Estática (95.8% de redução na rede)

2. Indexação Rápida LSH & Salvaguarda Linear (SSD)

Para buscar o padrão mais próximo em tempo real no navegador sem congelar a tela (pesquisar em 16.384 codewords para milhares de blocos), o JavaScript constrói em memória um índice LSH (Locality Sensitive Hashing). Ele agrupa os padrões em baldes (buckets) baseando-se nos tons do primeiro pixel do bloco.

Limiar Dinâmico e Fidelidade Visual: Se a busca no balde LSH retornar um erro SSD (Soma dos Desvios Quadráticos) superior a 1.000, o navegador realiza automaticamente um fallback para a Busca Linear Exaustiva. Isso garante fidelidade extrema nas transições e bordas diagonais de logotipos e textos, removendo ruídos indesejados.

3. Transmissão Híbrida: Primeira vs. Visitas Subsequentes

A grande sacada de largura de banda deste sistema é a separação lógica na rede:

  • Primeira Visita: O navegador baixa a página, o payload de índices da imagem e o dicionário de padrões visual (Cérebro, ~768 KB). O cérebro é armazenado localmente em formato Base64 no localStorage.
  • Visitas Subsequentes: O navegador faz uma chamada ultra-rápida do tipo HTTP HEAD para verificar a data de modificação e tamanho do dicionário. Se o dicionário não foi alterado no servidor, o navegador carrega a versão armazenada no localStorage e faz o download de apenas ~1 KB de índices (gzipped CROM) para desenhar a imagem!
Redução de Banda em Visitas Subsequentes: Economia de ~77% a 90% em comparação ao formato WebP

4. Processamento Concorrente Client-Side & SQL

O processamento ocorre totalmente na máquina do cliente, aliviando o uso de CPU do servidor:

  1. Você carrega uma imagem no navegador. O Javascript a quantiza localmente utilizando LSH.
  2. O payload de índices binários gerados localmente é codificado em Base64 e enviado via rota JSON POST para gravação direta no SQLite.
  3. O navegador baixa a imagem do banco para verificação e desenha o resultado em um elemento Canvas HTML5 escrevendo os pixels de forma contígua na memória do canvas, atingindo performance de renderização de frações de milissegundo.
